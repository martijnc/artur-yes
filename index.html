Abstract

This document describes a mechanism which allows developers to restrict the ways in which their pages can execute script, with the goal of mitigating cross-site scripting vulnerabilities.

1. Introduction

Cross-site scripting is a problem for developers. Despite the existence of context-aware escaping mechanisms in templating systems, content injection continues to be one of of the most common attack vectors.

Content Security Policy [CSP] aims to reduce the risk of these attacks, and is widely supported by browsers. It has proven to be difficult to deploy, however, as the syntax is complicated, the overriding rules are confused, and the feature itself serves several different purposes. Recent research has shown flaws in real-world deployments result in policies that are ineffective at directly preventing cross-site scripting in the overwhelming majority of cases [CSP-IS-DEAD]. That research, and related discussions, has lead to new approaches within CSP (like those described in Content Security Policy Level 3 Â§strict-dynamic-usage) which seem promising, but the syntax ends up being somewhat verbose and the behavior confusing due to backwards compatibility concerns.

This document proposes a policy language focused strictly on XSS protection, pulling in a subset of CSPâs functionality, along with existing functionality such as that supported by the X-XSS-Protection HTTP response header. The goal is to provide the same XSS mitigation guarantees as a verbose and complicated CSP, in a pithy and comprehensible package.

MegaCorp, Inc. wishes to defend against cross-site scripting. It can do so by generating a unique string for each page load, and including that string both in an XSS-Protection header and in the script elementâs nonce attribute, as follows: 
XSS-Protection: { "nonce": "abcdefg" }

<!-- This script will execute. -->
<script src="script.js" nonce="abcdefg"></script>

<!-- This script wonât (no nonce). -->
<script src="script.js"></script>

Note: This will also have the happy side-effect of enabling the user agentâs built-in XSS protections in "blocking" mode rather than the user agentâs default "filtering" mode. Hooray!

2. Framework

This specification uses terminology from [RFC5234] and [HTTP-JFV] to define its syntax expectations, and relies heavily on [CSP] (which itself relies on [FETCH] and [HTML]) for enforcement.

2.1. Policy

An XSS protection policy defines a set of behaviors which are to be applied to a Window or WorkerGlobalScope.

Each policy has a nonce which is a string. Its value is the empty string unless otherwise specified.

Each policy has a hash set which is a set of strings. Its value is the empty set unless otherwise specified.

Each policy has a reporting group which is a string. Its value is the string "default" unless otherwise specified.

Each policy has a reflection policy which is one of "ignore", "filter", or "block". Its value is "block" unless otherwise specified.

2.2. Algorithms

2.2.1. Obtain a set of XSS protection policies for response.

Given a response (response), this algorithm returns a sequence of XSS protection policy objects. If no valid policy is found for response, the returned sequence will be empty.

  1. Let policies be an empty sequence.

  2. Let value be the result of parsing "XSS-Protection" in responseâs header list.

  3. If value is null, return policies.

  4. Let list be the result of executing the algorithm defined in Section 4 of [HTTP-JFV] on value. If that algorithm results in an error, return policies.

  5. For each item in list:

    1. Let policy be a new XSS protection policy.

    2. If item has a hash member whose value (value) is a sequence<DOMString>, then for each digest in value:

      1. If digest matches the ABNF grammar hash-algorithm-hash-source (as defined in [CSP]), then add digest to policyâs hash set.

    3. If item has a nonce member whose value (value) is a DOMString that matches the ABNF grammar base64-value (as defined in [CSP]), then set policyâs nonce to value.

    4. If item has a reflection member whose value (value) is a valid ReflectedXSSPolicy, then set policyâs reflection policy to value.

    5. If item has a report member whose value (value) matches the token grammar (as defined in [RFC7230]), then set policyâs reporting group to value.

    6. Append policy to policies.

  6. Return policies.

3. The XSS-Protection HTTP Response Header

The XSS-Protection HTTP response header field is the delivery mechanism for the mitigations this document offers. The headerâs value is a serialized XSS protection policy represented by the following ABNF [RFC5234]:

XSS-Protection = json-field-value

XSS-Protection: { "nonce": "abcdefg", "reflection": "block", "report": "group1" }

This header will have more or less the same effect as delivering the following headers:

Content-Security-Policy: script-src 'strict-dynamic' 'nonce-abcdegf';
                         object-src 'none';
                         report-to group1;
X-XSS-Protection: 1; mode=block; report=group1;

A server MAY send different X-XSS-Protection header field values with different representations of the same resource.

The headerâs value is interpreted as an array of JSON objects, as described in Section 4 of [HTTP-JFV]. Detailed parsing instructions are found in Â§2.2.1 Obtain a set of XSS protection policies for response., but the syntax boils down to a representation of the following dictionary type:

enum ReflectedXSSPolicy {
  "ignore",
  "filter",
  "block"
};

dictionary XSSProtectionPolicy {
  sequence<DOMString> hash;
  DOMString nonce;
  ReflectedXSSPolicy reflection = "block";
  DOMString report = "default";
};

The following subsections define the behavior of each of the known members the headerâs value defines. Future versions of this document may define additional members; user agents MUST ignore unknown members when parsing the header.

3.1. The hash member

The hash member defines a set of digests which specify specific script content which is allowed to execute on a given page.

If present, the memberâs value MUST be a sequence of DOMStrings, and each MUST adhere to the hash-source grammar specified in Content Security Policy Level 3 Â§framework-directive-source-list. Any item in the sequence that does not match that grammer MUST be ignored.

MegaCorp, Inc. wishes to ensure that injected script elements do not execute. They can do so by executing a secure hash function (like those specified in [SHA2]) on each script they wish to execute, and including the resulting list of digests in an XSS-Protection header. These digests either match the inline content of a script element or event handler, or match a script elementâs integrity attribute: 
XSS-Protection: { "hash": [ "sha256-abcd...", "sha256-zyx..." ] }

...

<script src="script.js" integrity="sha256-abcd..."></script>
<script>
  // Content which hashes to abcd... goes here.
</script>
    
<a onclick="// Content which hashes to zyx... goes here.">Click!</a>

3.2. The nonce member

The nonce member defines a cryptographic nonce ("number used once") which can be used to ensure that only script elements whose nonce attributes contain a specific value are allowed to execute script for a given page.

If present, the memberâs value MUST be a DOMString, and MUST adhere to the base64-value grammar specified in Content Security Policy Level 3 Â§framework-directive-source-list. If the memberâs value does not match that grammar, the value MUST be ignored.

MegaCorp, Inc. wishes to ensure that injected script elements do not execute. They can do so by generating a unique string for each page load, and including that string both in an XSS-Protection header and in the script elementâs nonce attribute, as follows: 
XSS-Protection: { "nonce": "abcdefg" }

...

<script src="script.js" nonce="abcdefg"></script>

3.3. The reflection member

The reflection member controls the behavior of any built-in mechanisms a user agent provides to actively defend against reflected cross-site scripting.

The known values for this member are as follows:

ignore

Disables the user agentâs active defenses against XSS.

filter

Enables the user agentâs active defenses against XSS, filtering reflected script from the page. This is the default behavior for most user agents that implement such protections.

block

Enables the user agentâs active defenses against XSS, blocking the page entirely if reflected script is detected. Generally speaking, this is safer than filter, and we use it as the fallback when an XSS-Protection header is delivered without specifying a behavior.

User agents MAY choose not to support this attribute if they do not implement such heuristics.

MegaCorp, Inc. wishes to ensure that the user agentâs built-in XSS protection mechanisms are active, but does not want the browser to simply filter out suspected reflections. They prefer to block the page load entirely. They can do so with the following header: 
XSS-Protection: { "reflection": "block" }

3.4. The report member

The report member defines the reporting group to which violation reports will be sent [REPORTING].

The memberâs value MUST be a DOMString. If an empty string is provided, reporting will be disabled.

Note: Reporting to the group named "default" is enabled unless otherwise specified.

MegaCorp, Inc. wishes to receive violation reports when script execution is blocked by the XSS protection policy it specifies. It can do so by sending a report member along with the policy, and specifying a reporting endpoint, as follows: 
XSS-Protection: { "nonce": "abcdefg", "report": "group1" }
Report-To: { "url": "https://example.com/report", "group": "group1", "max-age": 10886400 }

4. Integrations

The XSS-Protection header is little more than syntactic sugar on top of existing processing algorithms for [CSP] and X-XSS-Protection. This section explains the ways in which the XSS protection policy is enforced for a given context:

4.1. Integration with Fetch

  1. When executing Content Security Policy Level 3 Â§set-response-csp-list (currently as step 3.3.4 of HTTP fetch, and step 12 of HTTP-network fetch), Fetch should also execute the algorithm defined in Â§4.1.1 Populate responseâs CSP list from its XSS protection policy..

4.1.1. Populate responseâs CSP list from its XSS protection policy.

Given a response (response), this algorithm modifies its CSP list to enforce the XSS protection policy it specifies.

  1. For each policy in the result of executing Â§2.2.1 Obtain a set of XSS protection policies for response. on response:

    1. If policyâs nonce is an empty string and policyâs hash set is empty, then skip the remaining substeps and move to the next policy.

    2. Assert: policyâs nonce is either an empty string or matches the base64-value grammar specified in [CSP]. Likewise, policyâs hash set is either empty or all its values match the grammar "hash-algorithm-hash-source". Further, policyâs reporting group is a string that matches the token grammar defined in [RFC7230].

    3. Let csp be a new Content Security Policy with an empty directive set, and a disposition of "enforce".

    4. Let script-src be the string "'strict-dynamic'".

    5. If policyâs nonce is not an empty string, append to script-src the result of replacing "%s" in the string "'nonce-%s' " with policyâs nonce.

    6. For each digest in policyâs hash set:

      1. Append to script-src the result of replacing "%s" in the string "'%s' " with digest.

    7. Add a new directive to cspâs directive set whose name is "script-src", and whose value is script-src.

    8. If policy has a reporting group (group) which is not an empty string, then add a directive to csp whose name is "report-to", and whose value is group.

    9. Append policy to responseâs CSP list.

4.2. Integration with HTML

  1. Each Document has a reflected XSS policy, which is one of null, "ignore", "filter", or "block". The value is null unless otherwise specified.

  2. When executing Content Security Policy Level 3 Â§initialize-document-csp (currently step 8 of the initializing a new Document object algorithm), HTML should also execute the algorithm defined in Â§4.2.1 Initialize a Document's reflected XSS policy..

4.2.1. Initialize a Document's reflected XSS policy.

Given a Document (document), and a response (response), this algorithm sets documentâs reflected XSS policy based on responseâs XSS-Protection and X-XSS-Protection headers.

  1. Let list be an empty sequence.

  2. Append the result of executing Â§4.2.2 Parse responseâs X-XSS-Protection header value. on response to list.

  3. For each policy in the result of executing Â§2.2.1 Obtain a set of XSS protection policies for response. on response, append policyâs reflection policy to list.

  4. Let result be null.

  5. For each item in list:

    1. If item is "block", set result to "block".

    2. If item is "filter", and result is not "block", set result to "filter".

    3. If item is "ignore", and result is not "block" or "filter", set result to "ignore".

  6. Set documentâs reflected XSS policy to result.

4.2.2. Parse responseâs X-XSS-Protection header value.

Given a response (response), this algorithm returns null, "ignore", "filter", or "block" based on the value (or absence) of its X-XSS-Protection header:

  1. Let value be the result of parsing "X-XSS-Protection" in responseâs header list.

  2. If value is null, return null.

  3. Let list be the result of strictly splitting value on U+003B SEMICOLON (";").

  4. Let result be "filter".

  5. Let processedFirstItem be false.

  6. While list is not empty:

    1. Let item be the result of popping the first item off of list.

    2. Strip leading and trailing whitespace from item.

    3. If processedFirstItem is false:

      1. If the first character of value is "0", return "ignore".

      2. If the first character of value is not "1", set result to "filter".

      3. Set processedFirstItem to true.

      Otherwise:

      1. If value is "mode=block", set result to "block".

  7. Return result.

5. Implementation Considerations

5.1. Error Messages

The XSS-Protection mechanism is defined almost entirely in terms of [CSP]. This is intentional, as it means that user agents can easily reuse all the existing machinery theyâve built up for that purpose. It might be confusing for developers, however, especially if, in the distant future, CSP becomes an esoteric thing that no one uses on its own. Error messages, in particular, might be confusing if they contain the synthesized CSP headers rather than the headers the developer actually sent.

That is, violation report console messages might look something like "Refused to execute inline event handler because it violates the following Content Security Policy directive: 'script-src 'strict-dynamic' 'nonce-abcdefg' ...". That is technically accurate (the best kind of accurate), but not particularly helpful.

User agents SHOULD ensure that the source of the policy is clear when triggering violations.

Perhaps we should even extend the SecurityPolicyViolationEvent and corresponding violation report?
